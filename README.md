# IEEE-CIS Fraud Detection

## Kaggle-ის კონკურსის მოკლე მიმოხილვა
ეს პროექტი მუშაობს Kaggle-ის კონკურსზე https://www.kaggle.com/competitions/ieee-fraud-detection. მონაცემები შეიცავს 590,540 ფინანსური ტრანზაქციის ინფორმაციას, სადაც 400+ ფიჩერი აღწერს ტრანზაქციისა და მომხმარებლის სხვადასხვა მახასიათებელს. target ცვლადი არის `isFraud`, რომელიც უნდა ვიწინასწარმეტყველოთ. პროექტის მიზანია სხვადასხვა მოდელებისა და ექსპერიმენტების ჩატარება, რათა დავინახოთ, რომელი მიდგომა იძლევა საუკეთესო შედეგს თაღლითობის აღმოჩენაში.

## მიდგომა პრობლემის გადასაჭრელად
თავდაპირველად დავაკვირდი მონაცემებს და მათში არსებულ გამოტოვებულ მნიშვნელობებს. ჩავატარე სხვადასხვა ექსპერიმენტი სამ არქიტექტურაზე:
1. **Logistic Regression** — საბაზისო წრფივი მოდელი, სხვადასხვა რეგულარიზაციით
2. **Random Forest** — ხეების ანსამბლი, სხვადასხვა სიღრმითა და კლასების დაბალანსებით
3. **XGBoost** — gradient boosting, ჰიპერპარამეტრების ტუნინგით

ყოველი ეტაპი დავლოგე MLflow-ის მეშვეობით DagsHub-ზე, რათა შემედარებინა მეტრიკები (ROC-AUC).

## EDA
მონაცემების გაანალიზებისას აღმოჩნდა, რომ ზოგიერთ სვეტში მონაცემების 90%-ზე მეტი გამოტოვებული იყო. ასევე შევამჩნიე მძიმე კლასური დისბალანსი — ტრანზაქციების მხოლოდ ~3.5% არის თაღლითური. ამ ორი გამოწვევის გადასაჭრელად მოდელების მიხედვით განსხვავებული სტრატეგიები გამოვიყენე.

## მონაცემთა გამოყოფა
მონაცემები `train_test_split`-ით შემთხვევითად გავყავი 80/20 შეფარდებით, `stratify=y`-ით, რათა კლასური პროპორცია შენარჩუნებულიყო.

## გასუფთავების მიდგომები

პროექტში გამოვიყენე განსხვავებული მიდგომები თითოეული მოდელისთვის, იმის მიხედვით, თუ როგორ ეპყრობა თითოეული ალგორითმი გამოტოვებულ მნიშვნელობებს.

**XGBoost — NaN-ების შენარჩუნება**
XGBoost-ისთვის გამოვიყენე მხოლოდ 90%-ის ზღვარი ხმაურიანი სვეტების ასაცილებლად, დანარჩენი NaN-ები კი შევინარჩუნე. XGBoost-ს აქვს ჩაშენებული მექანიზმი, Sparsity-Aware Split Finding, რომელიც NaN მნიშვნელობებს პირდაპირ ამუშავებს სპლიტების დროს — ამიტომ ხელოვნური შევსება არა მხოლოდ ზედმეტია, არამედ შეიძლება მოდელს ყალბი სიგნალები მიაწოდოს.

**Logistic Regression და Random Forest — SimpleImputer**
წრფივი და ხის მოდელები (sklearn-ის იმპლემენტაციები) ვერ ამუშავებენ NaN მნიშვნელობებს. ამიტომ გამოვიყენე:
- რიცხვითი სვეტებისთვის: `SimpleImputer(strategy='median')` — მედიანა მდგრადია გამოტოვებული მნიშვნელობების მიმართ.
- კატეგორიული სვეტებისთვის: `SimpleImputer(strategy='constant', fill_value='missing')` — ცალკე კატეგორია, რომელიც ასახავს ინფორმაციის არარსებობას.

გარდა ამისა, 90%-ზე მეტი გამოტოვებული მნიშვნელობის მქონე სვეტები წავშალე ხმაურის შესამცირებლად. ამ ზღვარმა 12 სვეტი წაშალა და დარჩენილ მოდელს შესაძლებლობა მისცა, ფოკუსირებულიყო უფრო სანდო ფიჩერებზე, რაც ვალიდაციის სტაბილურ AUC-ს იძლეოდა.

## Feature Selection

**მუდმივი სვეტების ამოშლა (Constant Feature Removal)**
გასუფთავებისა და ფიჩერების ინჟინერიის შემდეგ შევამოწმე, ჰქონდა თუ არა რომელიმე სვეტს მხოლოდ ერთი უნიკალური მნიშვნელობა:
```python
constant_cols = [col for col in X.columns if X[col].nunique() <= 1]
X.drop(columns=constant_cols, inplace=True)
```
ასეთი სვეტები სრულიად უსარგებლოა მოდელისთვის — თუ ყველა მწკრივი ერთსა და იმავე მნიშვნელობას შეიცავს, სვეტი ვერ განასხვავებს თაღლითურ ტრანზაქციას ჩვეულებრივისგან. ამათი შენარჩუნება მხოლოდ ზედმეტ ხმაურს დაამატებდა.

**სვეტების ამოშლა გამოტოვებული მნიშვნელობების ზღვარით**
90%-ზე მეტი გამოტოვებული მნიშვნელობის მქონე სვეტები წავშალე. სვეტი, სადაც მონაცემების 90%+ აკლია, ვერ მიაწვდის მოდელს საკმარის ინფორმაციას და მხოლოდ ხმაურს შეიტანს.

## Feature Engineering

## Pipeline და Preprocessing

გამოვიყენე `Pipeline` და `ColumnTransformer`, რათა მონაცემთა დამუშავება (Preprocessing) და მოდელის წვრთნა გაერთიანებულიყო ერთიან სტრუქტურაში. ეს მიდგომა უზრუნველყოფს, რომ:
- test set-ზე ზუსტად იგივე ტრანსფორმაციები გამოიყენება, რაც train set-ზე
- მოდელი შეიძლება პირდაპირ დაუმუშავებელ მონაცემებზე გაეშვას
- data leakage-ის რისკი მინიმალურია

კატეგორიული ცვლადებისთვის გამოვიყენე `OrdinalEncoder` `handle_unknown='use_encoded_value'`-ით, რათა test set-ში უცნობი კატეგორიები -1-ად დაკოდირდეს შეცდომის ნაცვლად. XGBoost-ისთვის მოდელად გამოვიყენე GPU-ს მხარდაჭერა (`tree_method='hist'`), რამაც მნიშვნელოვნად დააჩქარა ტრენინგი Kaggle-ის T4 GPU-ზე. Logistic Regression-ისთვის დამატებით გამოვიყენე `StandardScaler`, რადგან წრფივი მოდელები მგრძნობიარეა ფიჩერების სკალის მიმართ.

## ტრენინგი

### Logistic Regression (საბაზისო მოდელი)
პირველ რიგში გამოვიყენე წრფივი მოდელი, როგორც მინიმალური შესრულების ორიენტირი.

| ექსპერიმენტი | Val AUC |
|---|---|
| LR_baseline | 0.8559 |
| LR_strong_regularization (C=0.01) | 0.8620 |
| LR_balanced | 0.8629 |
| LR_weak_regularization (C=10.0) | 0.8629 |
| **LR_ridge** | **0.8629** |

`class_weight='balanced'` პარამეტრმა გააუმჯობესა შედეგი, რადგან მოდელი ითვალისწინებს კლასების დისბალანსს. რეგულარიზაციის სიძლიერეს (C პარამეტრი) მნიშვნელოვანი გავლენა არ ჰქონია — ეს ნიშნავს, რომ პრობლემა ძირითადად არაწრფივია და წრფივი მოდელი ვერ იჭერს საჭირო ინტერაქციებს.

**საუკეთესო მოდელი:** `LR_ridge` — val AUC **0.8629**

### Random Forest
შემდეგ გამოვიყენე დამოუკიდებელი გადაწყვეტილების ხეების ანსამბლი.

| ექსპერიმენტი | Val AUC |
|---|---|
| RF_baseline (100 ხე, სიღრმე=12) | 0.8760 |
| ExtraTrees (100 ხე, სიღრმე=12) | 0.8500 |
| RF_deeper (200 ხე, სიღრმე=20, balanced) | 0.9268 |
| RF_deeper_300trees (300 ხე, სიღრმე=20, balanced) | 0.9273 |
| RF_hyperparameter_tuning | 0.8839 |
| **RF_Final_Hybrid** (300 ხე, სიღრმე=25, balanced) | **0.9302** |

`class_weight='balanced'` იყო ყველაზე გავლენიანი ცვლილება (+0.05 AUC). ხეების რაოდენობის 200-დან 300-მდე გაზრდამ მინიმალური სარგებელი გამოიღო (0.9268 → 0.9273), რაც ნიშნავს რომ მოდელი 200 ხეზე უკვე კონვერგირდება. ჰიპერპარამეტრების ტუნინგი ნაკლებ ეფექტური გამოვიდა, რადგან ძებნა მხოლოდ მონაცემების 10%-ზე ჩატარდა RAM-ის შეზღუდვის გამო.

**საუკეთესო მოდელი:** `RF_Final_Hybrid` — val AUC **0.9302**

### XGBoost
ბოლოს გამოვიყენე გრადიენტული boosting GPU-ს აჩქარებით.

| ექსპერიმენტი | Val AUC |
|---|---|
| XGBoost_baseline (500 ხე, სიღრმე=6) | 0.9532 |
| XGB_slower_lr (1000 ხე, lr=0.05) | 0.9527 |
| XGB_scale_pos (scale_pos_weight=29) | 0.9565 |
| XGB_regularized (subsample+L1+L2) | 0.9562 |
| XGB_deeper (500 ხე, სიღრმე=8) | 0.9669 |
| **XGB_tuned** (700 ხე, სიღრმე=10, დაატუნინგებული) | **0.9723** |

სიღრმის გაზრდა 6-დან 8-მდე იყო ყველაზე მნიშვნელოვანი ერთი პარამეტრის ცვლილება (+0.014 AUC). ჰიპერპარამეტრების ტუნინგმა კიდევ უფრო გააუმჯობესა შედეგი — საუკეთესო კომბინაცია: ნელი სწავლის სიჩქარე (0.05) მეტ ხეებთან ერთად (700) და ძლიერი რეგულარიზაცია (reg_alpha=0.5, reg_lambda=2).

**საუკეთესო მოდელი:** `XGB_tuned` — val AUC **0.9723**

### შედეგების შედარება

| მოდელი | საუკეთესო Val AUC |
|---|---|
| Logistic Regression | 0.8629 |
| Random Forest | 0.9302 |

საუკეთესო შედეგი აჩვენა XGBoost-მა. Random Forest-მა მნიშვნელოვნად გაუმჯობესა Logistic Regression-ის შედეგი, რაც ადასტურებს, რომ ამ ამოცანისთვის არაწრფივი მოდელები უფრო ეფექტურია. XGBoost-ის უპირატესობა განპირობებულია boosting-ის თანმიმდევრული შეცდომების კორექციის მექანიზმით, რაც განსაკუთრებით ეფექტურია რთული ფიჩერური ინტერაქციების მქონე თაღლითობის მონაცემებზე. Logistic Regression კი გამოდგა მნიშვნელოვან საბაზისო ხაზად — ნებისმიერი მოდელი, რომელიც 0.8629-ს ვერ სჭარბობს, პრაქტიკულ სარგებელს ვერ მოიტანს.
